1. Project Purpose & Overview

MediScan AI is a comprehensive, end-to-end web application powered by Classical Machine Learning. The primary objective of this project is to provide a preliminary disease prediction based on a user's selected symptoms and recommend relevant medical information.

Key features of the application include:

  • Symptom Analysis: Users can select from 132 distinct symptoms.
  • Severity Assessment: Automatically calculates a severity score and provides a badge (Mild, Moderate, Severe) based on the chosen symptoms.
  • Disease Prediction: Predicts the most likely condition from 41 possible diseases using trained machine learning models, displaying a confidence chart for the top 5 predictions.
  • Actionable Insights: Provides a plain-English description of the predicted disease along with recommended precautionary steps.
  • Live Drug Recommendations: Integrates dynamically with the OpenFDA API to fetch and display real-world drug recommendations, including brand names, generic names, manufacturers, and intended purposes.
Disclaimer: The project was developed strictly for educational purposes and is not a substitute for professional medical advice.

3. Machine Learning Models & Training

The foundation of the application relies on the Disease Symptom Description Dataset (originally from Kaggle: itachi9604/disease-symptom-description-dataset). The dataset maps 132 binary symptom indicators to 41 unique disease classes.

Training Pipeline

  1. Data Preprocessing: Data was loaded, cleaned (stripping whitespace, filling NaNs), and labels were encoded using LabelEncoder.
  2. Splitting: Data was split into 80% training and 20% testing sets using stratified sampling to maintain class balance.
  3. Model Evaluation: Four distinct classical machine learning models were trained and evaluated:
    • Decision Tree: (Gini criterion, max depth 10)
    • Random Forest: (200 estimators, max depth 15)
    • XGBoost: (200 estimators, learning rate 0.1)
    • Support Vector Machine (SVM): (RBF kernel, probability=True)

Results & Selection

The models performed exceptionally well on this structured dataset. The Random Forest classifier emerged as the best performing model, achieving an accuracy and weighted F1-score of ~1.000 (100%).

The pipeline automatically saves the best model (best_model.pkl), the label encoder, and the feature column structure for use in the inference application.

4. Application Architecture (Gradio UI)

The user interface was built using Gradio, providing a sleek, reactive, and user-friendly medical dashboard.

  • Input Section: Features a searchable checkbox group for selecting symptoms. A live listener triggers a severity score update immediately upon selection.
  • Inference Engine: Upon submission, the app constructs a binary feature vector corresponding to the selected symptoms and passes it to the pre-trained Random Forest model using predict_proba().
  • Output Display: The results are formatted into clean Markdown cards, displaying the predicted disease, severity level, description, precautions, and a dynamically generated Matplotlib horizontal bar chart showing the top 5 predictions' confidence percentages.
  • OpenFDA Integration: The app concurrently makes an HTTP request to the OpenFDA API, querying the indications_and_usage field matching the predicted disease to return up to 3 medication suggestions.

5. Challenges Encountered & Resolutions

Throughout the development lifecycle, several critical technical hurdles were encountered and systematically resolved:

1. Python 3.14 Compatibility & Compilation Errors

  • Issue: The local environment was initially running the newly released Python 3.14. Essential packages like scikit-learn and xgboost lacked pre-built binary wheels for 3.14, forcing pip to attempt compiling them from C/C++ source. This failed due to missing MSVC build tools.
  • Resolution: Shifted the project to Python 3.12. We utilized the uv package manager to instantly provision a clean Python 3.12 virtual environment, allowing all packages to be installed seamlessly from pre-built wheels.

2. Dataset Availability Issues

  • Issue: The planned GitHub mirrors for the raw Kaggle CSV files returned HTTP 404 Not Found errors, and the Kaggle API requires authentication.
  • Resolution: Authored a custom generate_dataset.py script to programmatically synthesize the dataset. The script perfectly recreated the schema (132 symptoms, 41 diseases) by injecting core symptoms and random noise, generating ~1,230 robust training rows alongside severity and precaution metadata.

3. Windows Terminal Encoding Errors (CP1252)

  • Issue: Scripts crashed with UnicodeEncodeError on Windows when attempting to print stylized box-drawing characters and arrows to the terminal.
  • Resolution: Implemented a dynamic terminal re-configuration block: sys.stdout.reconfigure(encoding="utf-8", errors="replace") at the top of the scripts, overriding the default Windows CP1252 encoding.

6. Deployment to Hugging Face Spaces

The final phase involved deploying the application to the public internet via Hugging Face Spaces.

Deployment Steps

  1. Space Configuration: A new Gradio space was provisioned on Hugging Face. The README.md was configured with YAML frontmatter specifying the SDK, version, and entrypoint (app_file: app.py).
  2. Environment Adaptation: The Gradio launch configuration in app.py was adjusted to bind to 0.0.0.0 and dynamically disable the inbrowser=True flag when running on Hugging Face (detected via the SPACE_ID environment variable).
  3. Git Configuration & LFS: A comprehensive .gitignore was created to exclude local environments. A .gitattributes file was configured to track heavy model artifacts (*.pkl) and images using Git Large File Storage (LFS).
  4. Merge Conflicts & Push: During the initial push, Hugging Face had auto-generated files causing a merge conflict. We successfully performed a git pull --allow-unrelated-histories, resolved the conflicts favoring our local files using git checkout --ours, and completed the push.
